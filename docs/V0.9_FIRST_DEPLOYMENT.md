# v0.9 首次实施部署

> 适用版本：`0.9.1-1.20.1` · Capability schema 16 · 内容 schema v4 · 网络协议 7
>
> 历史说明：本页保留 0.9.1 首次迁移流程。当前 0.9.3 使用 Capability schema 17；正式恢复请优先使用 `scripts/restore_migration_backup.py`，并先执行 `scripts/check_save_rollback.py`。

## 为什么从存档安全开始

v0.9 第一次真实部署必须先保护既有 M1/M2 玩家和世界状态，再继续扩内容。当前五途径序列 9–7、试炼追踪、委托、调查链、余额、队伍账本、扮演历史和冷却都已进入长期存档；如果先改字段或批量增加数据，迁移错误会把可玩资产变成不可逆损失。因此 0.9.1 的第一纵切只处理备份、DataFix、未知数据保留和回归，不修改玩法数值。

## 自动保护链

1. Forge 在世界加载前触发 `ServerAboutToStartEvent`。
2. Mod 在世界目录 `project_mystery_backups/` 创建 `schema-16-<UTC 时间>` 快照。
3. 快照只复制 `level.dat*`、`playerdata/*.dat*` 和 Project Mystery 自有 SavedData，避免复制整个世界造成长时间阻塞。
4. 快照先写临时目录，再原子替换为正式目录；`schema-16.complete` 使同一 schema 的后续启动保持幂等。
5. 备份失败时服务器停止继续加载，禁止在无备份状态下迁移玩家存档。
6. Capability 加载时执行命名 DataFix 链，并把迁移前原始 NBT 写入最多三份 `migration_backups`。
7. 非法 ID、损坏特性 bundle、未来 schema 和无法安全解释的旧字段进入 `orphaned_entries`，不再静默删除。

自动快照不是完整服务器备份的替代品。正式服升级前仍应在停服状态额外复制整个世界目录。

## 版本迁移规则

| 来源 | 处理 | 与旧内容连接 |
|---|---|---|
| 无版本 / schema 0–14 | 先执行旧键名归一化，再进入 schema 15→16 | 保留知识、扮演计数和组织声望旧键 |
| schema 15 | 建立单层 legacy `CharacteristicBundle`，补齐扮演 v2 默认值 | 保留途径、序列、魔药品质、M1 试炼与 M2 委托/调查状态 |
| schema 16 | 原样读取，不重复迁移或重复备份 | 当前 0.9.x 正常路径 |
| 高于 schema 16 | 不执行降级修复；保存完整原始 NBT 为 `future_schema` 孤儿 | 防止使用旧 Mod 意外吞掉未来版本字段 |

迁移记录写入 `migration_history`。同一份已迁移存档再次保存、重登或重启时，不会重复生成特性层、备份或历史项。

## 部署步骤

1. 停止服务器，确认没有残留 Java 进程持有世界文件。
2. 额外复制完整世界目录到服务器外部备份位置。
3. 在干净仓库执行：

   ```bash
   ./gradlew clean build
   python scripts/run_server_smoke.py --timeout 180
   ```

4. 只替换 Project Mystery Mod JAR，不删除世界中的 `data/`、`playerdata/` 或 `serverconfig/`。
5. 首次启动检查日志是否出现“已创建 schema 16 迁移前备份”，并确认快照目录与 `manifest.properties` 存在。
6. 使用一个旧玩家登录，依次检查 `/pm status`、`/pm trial status`、`/pm party` 和 `/pm servercheck`。
7. 正常停服后再次启动，确认不再创建第二份 schema 16 快照，玩家途径、序列、特性、试炼和委托状态仍一致。

## 回滚

1. 立即停服，不要在错误版本上继续保存。
2. 换回升级前 JAR。
3. 从 `project_mystery_backups/schema-16-.../` 把 `level.dat*`、`playerdata/` 和 Project Mystery SavedData 恢复到世界目录；不要把新旧文件混合恢复。
4. 启动旧版本并验证玩家与队伍状态。
5. 保留失败版本日志、快照和原世界副本，用于修复后再次演练。
