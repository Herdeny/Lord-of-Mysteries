# v0.9 M0 内容图与迁移

## 内容图

`scripts/build_content_graph.py` 从当前仓库证据构图，而不是从设计目标推断完成度。当前节点来自：

- `docs/master/m0_content_catalog.json`
- `docs/master/m0_runtime_catalog.json`
- `src/generated/resources/data/lord_of_mysteries/pathway_catalog/`
- `src/main/resources/data/lord_of_mysteries/sequences/`
- `src/main/resources/data/lord_of_mysteries/commissions/`
- `src/main/resources/data/lord_of_mysteries/quests/`

当前基线为 126 节点、188 条有向关系、28 个未连入主闭环的孤儿节点和 20 条受知识门槛保护的剧透关系。孤儿并不自动代表错误；它们用于定位尚未进入配方、任务、掉落或知识链的迁移内容。

## 报告

| 报告 | 用途 |
|---|---|
| `pm-content-validation.json` | 汇总 schema、节点、关系和门禁结果 |
| `content_graph.json` | 提供完整节点和边，供后续可视化与工具读取 |
| `orphan_report.csv` | 列出未连入其他内容的定义 |
| `spoiler_leak_report.csv` | 检查高剧透内容是否经过知识门槛 |
| `compat_report.csv` | 区分已实现、计划和未支持兼容 |
| `localization_report.csv` | 校验中英文键覆盖 |
| `asset_report.csv` | 对照注册项、模型和资源 |

报告属于构建产物，不提交到版本库；每次 CI 都重新生成。

## M0 目录约束

`docs/master/m0_content_catalog.json` 与 `docs/master/m0_runtime_catalog.json` 共同构成 M0 迁移目录，覆盖：

- 10 个已注册物品
- 10 个当前状态/风险概念
- 5 个已存在配方
- 3 个知识门槛

新增目录项必须使用 schema v4 元数据，并通过运行时注册、翻译、模型、配方、前置关系和剧透门控检查。设计中存在但代码未注册的 ID 不得伪装为 `code_ready`。

## 存档迁移

本批完成：

- 玩家 Capability schema 15/16 → 17。
- schema 15 非凡者旧档自动回填 legacy `CharacteristicBundle`。
- 新字段参与复制、NBT 保存/读取和非法值修复。
- Capability Provider 在 Forge 生命周期结束时失效。
- schema 0/15/16→17 使用命名 DataFix 链，已迁移存档重复加载保持幂等。
- 世界加载前原子备份 `level.dat*`、玩家数据和 Project Mystery SavedData；备份失败时停止加载。
- 原始 Capability NBT、迁移历史和最多三份回滚载荷随玩家数据保留。
- 非法 ID、损坏 bundle、未来 schema 和无法安全解释的旧字段进入 `orphaned_entries`。
- Core/Knowledge/Social/Endgame 四区 dirty mask 捕获公共字段与集合原地修改。
- 登录、重生、维度切换使用独立核心摘要；每秒检查、5 秒校正且不发送知识全集。
- 9 项 Forge GameTest 验证 Capability Clone、schema 15→23 Provider 往返、future schema 隔离、精确特性载荷、M1 状态、调查板邻近、城市经济/暴露、五途径 7→6→5 重启和能力世界行为。
- `restore_migration_backup.py` 支持 dry-run、清单校验、恢复前安全快照、精确替换和路径穿越拒绝。
- `run_server_restart_matrix.py` 连续启动两次真实专服，校验世界种子与 schema marker 幂等。

M0 的内容数量、迁移、回滚、重启和自动化门禁均已关闭；后续持续补充真人死亡/重生/跨维度整体验收记录，并在 M2 扩展内容图。

首次部署与回滚操作见 `docs/V0.9_FIRST_DEPLOYMENT.md`。

## 内容贡献流程

1. 在主表或领域 JSON 中增加定义。
2. 填写 schema v4 来源、剧透、知识门槛、关系和实现状态。
3. 添加运行时注册、双语文本与模型/配方证据。
4. 运行生成器和内容图。
5. 审查孤儿、剧透、兼容、本地化与资产报告。
6. 更新 README、Pages、Wiki、CHANGELOG 和路线证据。
7. 通过 clean build 与专用服务器冒烟后再发布。
