# Project Mystery 专用服务器与多人一致性验证矩阵

> 适用版本：0.9.12-1.20.1
>
> 技术基线：Minecraft 1.20.1 · Forge 47.4.20 · Java 17
>
> 数据基线：Capability schema 19 · content schema v4 · SimpleChannel protocol 12

本矩阵用于验证 M1 连续性，并为 M2 多人内测提前冻结可重复的证据格式。自动冒烟通过不等于
真人两小时平衡验收通过；任何崩溃、复制、永久卡步、跨玩家串档或客户端越权均为阻断项。

## 自动门禁

```bash
python scripts/gen_datapack.py --check
python scripts/check_m1_playability.py
python scripts/check_m2_investigation.py
python scripts/check_resource_integrity.py
python scripts/check_save_rollback.py
python scripts/sync_project_metadata.py --check
./gradlew clean build
./gradlew runGameTestServer
python scripts/run_server_restart_matrix.py --timeout 180
```

`check_save_rollback.py` 会创建隔离测试世界，验证快照 dry-run、完整恢复、快照后文件清除、
恢复前安全备份与路径穿越拒绝。`run_server_restart_matrix.py` 会连续启动 Forge 专用服务器两次，
每次确认 4 个委托和 4 条任务链完成加载、进入 `Done`、运行 `/pm servercheck`、执行命令循环、
强制保存并干净停服；第二次还要求世界种子与 schema 19 marker 保持一致。GitHub Build 工作流执行同一矩阵。

## 开测前诊断

1. 使用新世界、空背包、普通人生存模式进入专用服务器。
2. 执行 `/pm doctor`；必须显示数据修复完成、委托/任务定义数量正确、协议为 12 且世界目标可计算。
3. 执行 `/pm trial reset` 后 `/pm trial start`；只有本轮需要完全清空时才使用 `reset`。
4. 记录世界种子、难度、玩家数、服务端配置、模组 JAR 哈希、开始时间和 Git commit。

## M1 连续性矩阵

| 场景 | 操作 | 必须保持 | 机器证据 |
|---|---|---|---|
| 新世界 | 首次登录并运行 `/pm doctor` | 手稿、罗盘、玩家数据和世界目标正常 | doctor 通过 |
| 退出重登 | `stop` 前后退出并重新进入 | 途径、序列、知识、委托、计时累计不丢失；离线时间不计入 | 重连 ≥1 |
| 死亡重生 | 携带活动任务死亡并重生 | Capability、任务步骤、统计按规则复制且无物品复制 | 死亡恢复 ≥1 |
| 跨维度往返 | 主世界→下界→主世界 | 玩家数据、任务、罗盘目标与计时一致 | 维度变化 ≥2 |
| 专服重启 | `stop` 保存、重新启动并登录 | 活动试炼、委托、记者/夜袭和仪式状态可恢复 | 服务重启 ≥1 |
| 世界时间回退 | 备份恢复或测试环境时间变小 | 计时不出现负数或异常暴增 | 状态数值正常 |
| 旧 schema | 从旧版玩家 NBT 启动 | 默认填充新字段，非法值被钳制，损坏引用被清理 | doctor 修复数 ≥0 |
| 仪式重启 | 调用/结算前停服再启动 | `RESOLVING` 安全回退重试，不永久卡死 | 祭坛可继续或取消 |
| 主持人离线 | 仪式中主持人离线 | 调用暂停；持续离线 60 秒后取消且不错误结算材料 | 祭坛状态回到安全态 |

结束时 `/pm trial verify` 必须显示九项核心目标 9/9、连续性门禁 4/4。随后仍需填写
[`M1_ACCEPTANCE_CHECKLIST.md`](M1_ACCEPTANCE_CHECKLIST.md) 的材料、战斗和节奏记录。

## M2 队伍矩阵

共享任务以原版记分板队伍作为当前临时组队边界。在线推进即时同步，已登记离线成员通过主世界
持久调查账本在上线时追赶；奖励、声望和冷却仍按玩家独立结算。

| 人数/状态 | 预期 |
|---|---|
| 1 人 | 完整单人流程不依赖队友，所有步骤可独立推进 |
| 2–4 人同队在线 | `shared_progress=true` 的目标同步推进并写入队伍账本；每名成员仍独立结算奖励 |
| 2–4 人不同队 | 不共享任何任务进度、记者 UUID 或夜袭波次 |
| 5 人以上同队 | 超过 `max_party=4` 时安全降级为单人，不产生部分共享或串档 |
| 队员中途加入 | 满足前置且无冲突委托时执行 `/pm party sync`，追赶到队伍最新步骤；不补发他人已结算奖励 |
| 队员中途离线 | 在线成员继续；已登记成员上线时自动追赶步骤、计数、记者、夜袭和路线状态 |
| 一人先结算 | 只标记该成员并发放其个人奖励；其他离线成员继续保留待结算状态 |
| 成员换队/退队 | 旧账本删除该 UUID，个人任务状态保留，旧队其他成员不受影响 |
| 超员期间个人结算 | 5 人安全降级期间仍按 UUID 与案件 ID 注销，不因当前队伍超限留下可重复奖励状态 |
| 已完成成员执行 `sync` | 不可重复委托拒绝再次加入；可重复委托必须通过冷却 |
| 损坏账本恢复 | 无效键、记者 UUID、完成账本和越界步骤被安全清理或拒绝 |
| 协调者离线 | 按 UUID 确定的新协调者接管护送/夜袭 tick，不重复生成波次 |
| 多队并行 | 追兵按队伍键隔离，一队击杀不得推进另一队波次 |

每组至少验证：接取、`/pm party`、中途 `sync`、放弃、证据拾取、三次威胁、记者 UUID、护送到达、
三波夜袭、离线追赶、个人最终结算、死亡、跨维度、重连、服务器重启、`/reload` 和超过人数上限降级。

## M2 证据推理与案件恢复矩阵

| 场景 | 操作 | 必须保持 |
|---|---|---|
| 空档案 | 无活动案件时打开调查板并执行 `/pm case analyze` | 推理页显示无案件；命令不给出伪造结论或奖励 |
| 未揭示关系 | 猜测一个当前未显示的关系 ID 提出假说 | 服务端拒绝；不写草稿、不改变压力、进度或奖励 |
| 假说误判与纠正 | 对已揭示关系提交错误立场并检验，10 秒后提交正确立场 | 首次压力 +4、负担 +1；正确检验后负担恢复 1；案件仍可继续 |
| 假说复议 | 错误检验后立即/等待 30 秒执行复议 | 立即复议被冷却拒绝；到期后清草稿并恢复 1 点负担 |
| 收集阶段 | 接案后只取得首批证据 | 阶段为“收集中”，完整度、已确认/矛盾/缺失数量与服务端进度一致 |
| 关联阶段 | 推进到至少一条关系两端都已发现 | 阶段为“关联中”，仅显示已发现关系，不提前泄露隐藏证据 |
| 结论就绪 | 收集案件所需关键证据 | 阶段为“可结论”，理论文本与服务端真值一致，仍不自动结算案件 |
| 真假矛盾 | 完成真假配方三项鉴定 | 真卷宗无矛盾；伪造卷宗至少一项矛盾，理论根据矛盾综合得出 |
| 下一步指引 | 每个步骤执行 `/pm case analyze` | 只返回一个与当前服务端步骤对应的明确行动，不接受客户端自报进度 |
| 委托纸丢失 | 活动案件中丢弃委托纸，在真实调查板附近执行 `/pm case recover` | 只补一份委托纸；重复执行不复制；远程执行拒绝 |
| 配方卷宗丢失 | 真假配方案第 2–4 步丢弃卷宗并在调查板附近恢复 | 只补一份与原案件接受时间和玩家 UUID 对应的确定性卷宗，隐藏真伪不改变 |
| 非适用物品 | 其他案件、错误步骤或物品仍在背包时执行恢复 | 不补发卷宗，不覆盖现有 NBT，不推进步骤，不发奖励 |
| 多人隔离 | 2–4 人分别查看推理或恢复 | 案件、证据、卷宗真伪和恢复结果按玩家隔离；共享进度仍遵守队伍账本规则 |
| 重连/重启 | 记录推理阶段后退出、重启并返回调查板 | 证据由持久任务状态重新推导，阶段和关系不倒退、不重复生成关键物品 |

人工记录至少包括：案件 ID、步骤、证据完整度、关系数、下一步文本、恢复前后背包截图、玩家 UUID、队伍键和重启前后结果。完整规则见 [`M2_EVIDENCE_REASONING_AND_RECOVERY.md`](M2_EVIDENCE_REASONING_AND_RECOVERY.md)。

## M2 动态案件轮换矩阵

| 场景 | 操作 | 必须保持 |
|---|---|---|
| 三日轮换 | 连续记录三个游戏日的 `/pm case rotation` | 三类案件各出现一次；同一天重复查询实例 ID 与八槽位完全一致 |
| 接案跨日 | 日落前接案，跨午夜后再次查看卷宗 | 活动案件沿用接案 tick，不被新一天预览替换 |
| 现场实体 | 加载卷宗指定结构并等待 100 tick 周期 | 生成一名当前主体、一名当前影响角色和一个可视物证；名称/职业/物品主题与卷宗槽位一致 |
| 可视物证入口 | 携带当前证物袋右键悬浮物证，再在另一轮右键附近方块并测试 `investigate field` | 三种入口均只在主世界目标 28 格内且实例/步骤匹配时推进；远程、空手或旧袋不推进 |
| 证人入口 | 右键指定标签 NPC，再杀死/卸载 NPC 测试替代 | 直接交互零费用并记录证词；NPC 不可用时 `field`/调查板 `desk` 仍能推进 |
| 档案入口 | 右键当日报纸，再丢弃报纸测试替代 | 直接交互记录档案；遗失后调查板 `desk` 仍能推进 |
| 证物袋同步 | 2–4 人分别持有本轮证物袋并由一人推进三轮证据 | 所有在线参与者阶段同步为服务端步骤；三证据后发光，不修改个人费用/压力 |
| 封存样本同步 | 完成现场后检查 2–4 人背包，再让一名成员中途同步 | 每人只取得当前主题的一件实例绑定样本；中途成员按权威步骤补齐，不生成其他主题 |
| 物品恢复 | 丢弃证物袋和封存样本后在调查板执行 `/pm case recover`，再重复执行 | 只补回当前实例、当前阶段及主题各一份；已有物品不复制；远程拒绝 |
| 结案与放弃 | 分别结算一轮和放弃一轮 | 当前实例证物袋与封存样本自动回收；旧实例或伪造 NBT 不能推进下一轮 |
| 替代费用 | 分别以余额 ≥6 和余额 <6 执行 `desk` | 前者只扣 6 便士；后者只加 3 点压力；均不重复推进同一步 |
| 正确结论 | 根据手段提交对应结论 | 完成任务链，结论路线写入复盘，回板后每名成员独立结算 |
| 错误结论 | 提交错误结论后连续尝试其他答案 | 增加 6 点压力并锁定 `reconsider`；未复议前不能穷举 |
| 误判恢复 | 在调查板执行 `rotation recover` 或统一 `case recover` | 状态变为 `recovered`，压力恢复 2 点，伪证关系变为确认矛盾并允许重试 |
| 重启恢复 | 在现场后、三证据后、误判后各保存重启 | 实例 ID、八槽位、步骤、接案 tick、共享结论状态均不漂移 |
| 多队隔离 | 两支队伍同日接取并交错推进 | 同日案件真相一致；步骤、费用、压力、假说、结算与冷却不跨队串写 |
| 跨日入队 | 队长前一日接案，成员次日加入已有队伍账本 | 新成员先采用账本接案 tick；主体、样本、真相和实例 ID 与队长一致 |
| 旧现场清理 | 所有人离线或结束案件后重新加载站点，再让活动成员返回 | 已加载旧主体/影响角色/展示物清理；活动实例返回时可从服务端状态重建且不掉落展示物 |
| 新轮假说 | 完成一轮后冷却结束再接取 | 上轮假说草稿被清除；最近复盘可查，不影响新实例关系 |

详细生成合同和命令见 [`M2_DYNAMIC_CASE_ROTATION.md`](M2_DYNAMIC_CASE_ROTATION.md)。

## 网络与数据安全矩阵

- 协议版本必须完全匹配；旧协议客户端不得静默接入。
- 14 个消息使用固定 ID，新增消息只能追加，禁止重排既有 ID。
- 调查板接取/放弃只能在真实调查板附近执行，远程重放和伪造案件 ID 必须被服务端拒绝。
- 核心摘要包不得打开状态 GUI，也不得携带完整知识集合；登录、重生和换维度后必须立即校正。
- 连续快速发送十类 C2S 意图时，服务端限流必须拒绝超频请求且不重复扣除/结算。
- 客户端伪造目标、距离、灵性、冷却、PvP 权限或任务步骤不得改变权威状态。
- `/reload` 后委托、任务、双语键、目标类型、合作人数和跨定义引用仍通过校验。
- `zh_cn` 与 `en_us` 均不得出现缺失翻译键或内部资源 ID 直出。
- 统一资源门禁必须验证全部 JSON、双语键、模型/纹理引用以及物品、方块、实体注册资源。
