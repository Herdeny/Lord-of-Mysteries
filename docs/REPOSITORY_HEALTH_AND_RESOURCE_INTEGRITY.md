# 仓库健康审计与资源完整性门禁

> 适用版本：0.9.18-1.20.1
>
> 技术基线：Minecraft 1.20.1 · Forge 47.4.20 · Java 17 · Gradle 8.14.5

本页记录 0.8.9 健康审计，并持续追加到 0.9.18 的 schema 23、正式城市街区、三职业经济、神秘暴露、六种世界事件、Pages 注册图鉴和 M3 五途径序列 6–5 合同。目标不是增加表面内容，
而是确保现有迁移资产在损坏数据、换队、重复同步、资源漂移和未来构建工具升级时仍能安全失败。

## 本轮修复

- 队伍成员离开或切换记分板队伍时，从旧调查账本移除 UUID；个人任务进度保留，不跨队串档。
- 放弃与结算按玩家 UUID、委托 ID 和任务链 ID 查找账本，不再依赖当前队伍人数或名称。
- 已完成不可重复委托不能通过 `/pm party sync` 再领奖；可重复委托同步必须通过冷却检查。
- 只有已登记且未结算成员可以推动权威账本，四人上限不能被离线旧成员或损坏 NBT 绕过。
- 无效队伍键、完成账本、损坏记者 UUID、越界步骤和目标计数会被丢弃或拒绝。
- 任务进度使用长整型安全累加，避免极端或损坏输入导致整数回绕。
- 委托与任务链的运行时解析和生成器统一校验非空、唯一性、等级、失败策略、人数、计数与溢出。
- Forge 入口改用构造上下文注入；Gradle 发布仓库移除旧 Maven DSL。`--warning-mode all` 已确认剩余提示来自 ForgeGradle 6 的 `downloadMcpConfig` / `reobfJar` 内部实现，而非项目脚本。
- 动态案件同站点使用稳定 25 槽布局，可视物证校验来源实例；日程统一读取主世界昼夜，跨维度与多队并行不串场。
- 真假配方卷宗绑定玩家与接取 tick；记者护送改为队伍专属实体并兼容旧公共记者存档迁移和拆队分叉。
- 委托书、卷宗、证物袋和样本满背包时不落地，结算/放弃回收当前绑定物；已登记队伍后来扩容不再中断原成员。
- 组织响应任务绑定玩家、案件实例、组织、联系人、周指令、关系分支、阶段和三至四日截止时间；错误 NPC、远程提交和重复提交不会推进或结算。
- 联系人关系账本最多 24 条，按案件实例去重；共同结案、履约、放弃和逾期可追溯，分支选择使用有界计数且不保存真实聊天。
- 响应过期、调查板主动放弃、损坏历史不匹配和非法旧载荷都会释放唯一任务槽，避免旧档或长期离线导致永久软锁。

## 统一资源门禁

运行：

```bash
python scripts/check_resource_integrity.py
```

当前门禁验证：

| 范围 | 当前基线 |
|---|---:|
| 可解析 JSON | 303 |
| 中英成对翻译键 | 1343 |
| Java/JSON 静态引用翻译键 | 514 |
| 模型 JSON | 90 |
| 注册物品 | 85 |
| 注册方块 | 5 |
| 注册实体 | 8 |

检查器会拒绝：

- 任意主资源或生成资源 JSON 语法错误；
- `en_us` / `zh_cn` 键集合漂移或静态引用缺失；
- 模型引用不存在的本 Mod 模型或纹理；
- 注册物品缺少模型/名称，注册方块缺少名称、blockstate 或模型，注册实体缺少名称。

`checkResourceIntegrity` 已加入 Gradle `check`，Build 与 Documentation Consistency 工作流也会独立执行。
`check_github_text_encoding.py` 会在 Issue/评论创建或编辑时通过 GitHub API 扫描全部协作文本，拒绝连续问号和 Unicode 替换字符，
因此代码、资源、文档与协作文本改动都不能绕过对应门禁。

## 自动验证基线

- 340 项 JUnit 单元/契约测试，要求失败、错误与跳过均为 0。
- 9 项 Forge GameTest 必须真实执行，验证 Capability Clone、schema 23 Provider 往返、future schema 隔离、精确特性载荷、M1 状态、调查板邻近门禁、队伍 SavedData、五途径 7→6→5 晋升重启与能力世界行为。
- M1/M2/M3 三份合同、正式街区 v2、12 项周指令、组织立场、三职业经济、暴露、世界事件、Pages 注册图鉴、资源完整性和 GitHub Issue/评论编码审计。
- `processResources` 固定使用 UTF-8；发布 JAR 内 `mods.toml` 的中文简介与当前 M2 描述不再受 Windows 系统代码页影响。
- Forge `clean build`、隔离回滚演练与两次真实专用服务器启动、`/pm servercheck`、命令循环、强制保存、干净停服。
- 服务端诊断输出协议/包数量、世界状态、队伍 SavedData、活动队伍数和活动成员数。
- Capability schema 23、内容 schema v4、网络协议 12；四区 dirty mask、权威快照、schema 0/15–22→23 DataFix、三职业班次、神秘暴露、结案/假说/动态案件/联系人/响应、世界快照、精确恢复、原始 NBT 备份和孤儿保留均进入自动验证。

## 仍需人工验证

- M1 两小时真实生存、材料节奏、伤害、灵性和风险恢复手感。
- 2–4 人真实网络下的带宽、tick、掉线、死亡、重启和多队并行。
- 真人客户端跨维度、死亡重生和断线整体验收。
- 四人八小时并行案件、跨日三职业经济、世界事件切换、暴露恢复与任务无锁死记录。
- 五途径序列 6–5 的专属晋升仪式、经济定位和无唯一最优途径真人矩阵。
- 队名删除后复用、管理员离线改队、正式邀请/队长权限和最终队伍 GUI。
- 原创像素资产、公开术语与第三方知识产权逐项登记。

自动门禁通过只表示现有实现满足机器可验证契约，不替代 `docs/M1_ACCEPTANCE_CHECKLIST.md` 与专服人工矩阵。
