# 专用服务器与多人一致性验证

> 当前版本：0.9.18-1.20.1 · Capability schema 23 · 内容 schema v4 · 网络协议 12

## 自动冒烟

```bash
python scripts/check_save_rollback.py
python scripts/run_server_restart_matrix.py --timeout 180
```

回滚脚本在隔离世界中验证 dry-run、完整恢复、快照后文件清除、恢复前安全备份与路径穿越拒绝。
重启矩阵连续启动 Forge 专用服务器两次，确认 4 个委托、4 条任务链完成加载并进入 `Done`，随后执行
`pm servercheck`、`list` 和 `save-all flush`；第二次必须保留相同世界种子、schema 23 marker、世界事件和前哨服务版本。Build 工作流会执行同一检查。

`./gradlew check` 还会运行 `scripts/check_m1_playability.py`，核对 120 分钟、9 项核心目标、7 个里程碑、三份 M1 魔药、
十二项保底补给、十个关键命令入口和中英文本地化；M2 合同保护调查板、真假配方、三解、持久队伍恢复、动态案件相关人/受影响者/可视物证/封存样本/证物袋/NPC/报纸交互与旧 13 步索引；
统一门禁保护 v0.9 设计源、126 节点/188 关系内容图、303 JSON、1343 对双语键、90 模型与全部注册资源；当前基线为 340 项 JUnit、9 项真实 Forge GameTest，并验证 schema 23、M1/M2/M3 合同、正式街区 v2、12 项周指令、组织立场、三职业经济、暴露、六事件、队伍隔离、五途径 7→6→5 重启往返和能力世界行为。

## M2 证据推理与恢复必测项

- 三案分别从“收集中”推进到“关联中”和“可结论”，完整度与关系只随服务端证据变化。
- `/pm case analyze` 在每一步给出唯一下一行动；无案件时不得生成结论。
- 委托纸只在活动案件且背包缺失时恢复；重复执行不得复制。
- 真假配方卷宗只在第 2–4 步恢复，恢复后的隐藏真伪必须与原案件一致。
- 交换玩家卷宗或使用旧/未绑定卷宗必须被服务端拒绝；满背包时委托书、卷宗、证物袋和样本不得掉到公共世界。
- `/pm case recover` 必须在真实调查板附近执行，远程请求不得改变背包、进度、余额或奖励。
- 2–4 人同队和不同队都要核对玩家级卷宗、证据、恢复结果不串档。

详细操作与记录格式见[仓库专服矩阵](https://github.com/Herdeny/Lord-of-Mysteries/blob/main/docs/DEDICATED_SERVER_TEST_MATRIX.md)和[证据关联与案件恢复](M2-Evidence-Reasoning-and-Recovery)。

动态案件还需逐项记录三日轮换、接案跨日、主世界/下界/末地日程一致、主体/受影响者/可视物证生成、同站点 25 槽布局、跨实例展示物拒绝、现场方块采样、四类样本同步、标签 NPC 证词、报纸档案、调查板补领/费用/压力替代、结案/放弃回收、联系人评级/关系账本、三类响应分支、正确组织 NPC 简报、调查板响应提交、三至四日过期/主动放弃、跨日入队、旧现场清理/重建、错误结论锁定、复议恢复、重启稳定与多队隔离。完整表格见[八槽位动态案件轮换](M2-Dynamic-Case-Rotation)。

`./gradlew runGameTestServer` 必须显示 9 项测试实际运行并全部通过；报告 `0 tests`、仅编译成功、缺少五途径晋升或能力场景测试都不视为有效门禁。

## M1 连续性

开测先执行 `/pm doctor`，再用 `/pm trial reset` 与 `/pm trial start` 创建干净记录。
`stop` 只结算在线时长，`resume` 或再次 `start` 会恢复保留记录。

测试过程中用 `/pm trial report` 对照营地 10 分钟、序列 9/8/7 为 30/60/90 分钟、身份/反思/城市生活为 100/110/120 分钟的阶段目标，
并记录首次特殊生物击杀、首次正收益扮演和首次 25 风险峰值。

`/pm trial verify` 要求：

- 九项核心目标 9/9：两小时、营地、序列 7、三次狩猎、两次有效扮演、风险峰值、身份绑定、反思和城市生活。
- 四项连续性 4/4：至少一次重连、一次专服重启、两次维度变化和一次死亡恢复。

机器证据不能替代真实两小时材料、战斗与数值平衡记录。

## M2 多人基础

- 同一原版记分板队伍中的 2–4 名玩家可共享 `shared_progress` 任务目标并写入持久账本。
- 记者护送按队伍生成独立实体，夜袭按 UUID 选出唯一在线协调者；波次、记者和路线状态持久化，旧公共记者自动迁移。
- 不同队伍互不共享；首次超过 `max_party=4` 时安全降级为单人。
- 已登记离线成员上线自动追赶，中途加入使用 `/pm party sync`；每名成员独立结算。
- 已登记 1–4 人账本后来扩容时原成员继续可玩，新第 5 人保持独立；换队/退队会分叉护送记者并清理旧关系。
- 正式队伍 GUI、管理员离线改队/队名复用人工矩阵和五途径多人负载矩阵尚未完成。

## M2 城市服务

- 新世界必须生成报社、事务所、警局三座 7×7 建筑；旧前哨幂等升级到 `service_version=2`，NPC 搬迁且重启不重复。
- 报社、事务所、巡警共享每日一次工作限制，工资、纸张成本、压力和暴露按当前世界事件精确结算。
- 暴露始终钳制为 0–100；schema 22→23 迁移保留所有既有案件、联系人和响应字段。
- 2–4 人并行工作/调查时，余额、背包、风险、暴露和日限按玩家隔离。

## M3 序列 6–5

- 五条途径都必须完成 7→6→5，并保持精确灵性上限和两层特性账本。
- 十份坩埚配方逐一测试正确顺序、温度、辅材与跨途径污染。
- 玩家催眠/束缚/窃取必须拒绝；PvP、Boss、掉落 owner、书本代数、村民每日限次和安全传送全部由服务端重验。
- 自动化状态仅为 `code_ready`；专属晋升仪式和四人互补平衡仍需真人记录。
