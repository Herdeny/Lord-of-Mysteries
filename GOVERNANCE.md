# Project Governance

## 原则

Project Mystery 采用维护者主导、公开讨论、Pull Request 审核的协作模式。
仓库现有实现是行为基准；设计文档用于规划，不自动覆盖已实现且已验证的规则。

## 角色

- **Contributors**：提交 Issue、Pull Request、文档、测试、翻译或原创资产。
- **Reviewers**：持续贡献并熟悉相关模块，可提供正式技术审核。
- **Maintainers**：管理路线图、版本、仓库权限、许可和最终合并决策。

当前代码所有者记录在 `.github/CODEOWNERS`。成为 Reviewer 或 Maintainer
依据持续贡献、判断质量、协作行为和项目需要，不以提交数量自动决定。

## 决策流程

- 缺陷修复和小改动：通过 Issue/PR 讨论，由相关代码所有者审核。
- 新系统、破坏性变更、依赖变更、存档格式变更：先提交提案，至少保留
  7 天讨论窗口，紧急安全修复除外。
- 许可、品牌、IP 边界、支持版本：必须由 Maintainers 一致同意。
- 无法达成一致时，优先选择可逆、最小范围且与当前实现兼容的方案。

## 发布流程

1. 里程碑中的阻塞项完成。
2. `main` 构建与测试通过。
3. 更新版本号、变更日志、许可/归属信息。
4. 创建带签名说明的 GitHub Release 和对应 Git tag。
5. 发布的二进制必须能定位到完全对应的公开源代码。

## 安全与移除

维护者可在不等待常规讨论窗口的情况下撤回泄露凭据、恶意代码、侵权素材或
高风险构建。相关决策会在不泄露敏感信息的前提下留下记录。
